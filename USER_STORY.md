# HH Applicant Tool — Полный User Story

---

## 1. Обзор проекта

**Название:** HH Applicant Tool  
**Тип:** CLI-утилита для автоматизации поиска работы на hh.ru  
**Язык:** Python 3.11+  
**Архитектура:** Модульная, плагин-ориентированная (Vertical Slice Architecture)  
**Лицензия:** Limited Non-Commercial License  

---

## 2. Целевая аудитория (Personas)

| Персонаж | Описание | Боли | Цели |
|----------|----------|------|------|
| **Активный соискатель (Senior/Lead)** | Опытный разработчик, ищущий новую позицию | Рутиная рассылка откликов, обновление резюме, потеря контактов при отказах | Автоматизировать рутину, не пропустить хорошие вакансии, сохранить контакты HR |
| **Массовый отклик-охотник** | Шлет сотни откликов в день | Лимиты HH (200/день), теневой бан, капчи, спам-фильтры | Обходить ограничения, масштабировать через профили/прокси |
| **DevOps / Самохостинг-энтузиаст** | Запускает на своём сервере/VPN | Сложность деплоя, управление секретами, мониторинг | Docker Compose, переменные окружения, несколько профилей |
| **AI-энтузиаст** | Хочет умную фильтрацию и генерацию сопроводительных писем | Ручной отбор вакансий, написание уникальных cover letter | LLM-фильтрация, AI-генерация писем, решение капчи |

## 3. Ключевые User Stories

### Epic 1: Авторизация и управление сессией

| ID | User Story | Приоритет | Критерии приёмки |
|----|------------|-----------|------------------|
| US-1.1 | Как пользователь, я хочу авторизоваться на hh.ru через email/телефон и пароль, чтобы получить доступ к API | **Must** | Успешная авторизация, сохранение access/refresh токенов в конфиге, поддержка 2FA через email/SMS |
| US-1.2 | Как пользователь, я хочу авторизоваться через кода из официального Android-приложения (client_id/secret), чтобы не палиться в нарушении ToS | **Must** | Работа с декомпилированными credentials, имитация официального приложения |
| US-1.3 | Как пользователь, я хочу решать капчу в терминале (kitty/sixel протоколы), чтобы авторизоваться на headless-сервере | **Must** | Отображение капчи в Kitty/Ghostty/Konsole, fallback на sixel |
| US-1.4 | Как пользователь, я хочу управлять несколькими профилями (аккаунтами HH), чтобы рассылать отклики от разных личностей | **Must** | Флаг `--profile`, изолированные конфиги/БД/токены на профиль |
| US-1.5 | Как пользователь, я хочу выходить из профиля (отзывать токен), чтобы безопасно сменить аккаунт | **Should** | Команда `logout`, очистка токенов |

### Epic 2: Поиск и отклик на вакансии

| ID | User Story | Приоритет | Критерии приёмки |
|----|------------|-----------|------------------|
| US-2.1 | Как пользователь, я хочу искать вакансии по поисковому запросу (с синтаксисом HH: `(Go OR Golang) NOT PHP`), чтобы находить релевантные позиции | **Must** | Параметр `--search`, поддержка полного синтаксиса HH |
| US-2.2 | Как пользователь, я хочу откликаться на рекомендованные вакансии без указания поискового запроса, чтобы охватывать максимальный пул | **Must** | Работа без `--search`, использование рекомендаций HH |
| US-2.3 | Как пользователь, я хочу исключать вакансии по стоп-словам (fullstack, junior, php), чтобы не тратить лимиты на нерелевантное | **Must** | Параметр `--excluded-filter`, разделение запятыми |
| US-2.4 | Как пользователь, я хочу фильтровать вакансии через AI (LLM), чтобы отсеивать мусор семантически, а не по ключевым словам | **Should** | Режимы: `strict`, `balanced`, `permissive`; отдельные API-ключи для фильтрации |
| US-2.5 | Как пользователь, я хочу генерировать уникальные сопроводительные письма через AI для каждой вакансии, чтобы повысить конверсию | **Should** | Шаблоны + LLM, отдельные настройки `openai_cover_letter` |
| US-2.6 | Как пользователь, я хочу делать dry-run рассылку, чтобы протестировать параметры без реальных откликов | **Must** | Флаг `--dry-run`, вывод в консоль без отправки |
| US-2.7 | Как пользователь, я хочу ограничивать количество откликов (`--per-page`, `--total-pages`), чтобы контролировать дневной лимит (200/день) | **Must** | Параметры пагинации, учёт лимитов |
| US-2.8 | Как пользователь, я хочу указывать регион поиска (`--area`), чтобы искать в конкретных городах/странах | **Must** | Поддержка area_id (Москва=1, Казахстан=40, Беларусь=2238) |
| US-2.9 | Как пользователь, я хочу автоматически решать капчи при отклике через AI (vision модели), чтобы не прерывать процесс | **Could** | Настройка `openai_captcha`, интеграция с playwright |

## 4. Нефункциональные требования

| Категория | Требование |
|-----------|------------|
| **Производительность** | Асинхронный HTTP-клиент (httpx), пул соединений, rate limiting (настраиваемый per AI-провайдер) |
| **Надёжность** | Retry с экспоненциальным бэкофом, идемпотентность операций, миграции БД (`migrate`) |
| **Безопасность** | Токены в конфиге (не в коде), маскирование в логах, поддержка прокси, изоляция профилей |
| **Кроссплатформенность** | Linux, macOS, Windows (в т.ч. WSL), рутованные Android-телефоны |
| **Наблюдаемость** | Структурированное логирование (`-v`, `-vv`), цвета, прогресс-бары (rich) |
| **Совместимость** | Python 3.11+, современные зависимости (pydantic v2, sqlalchemy 2.0, textual) |

## 5. Архитектурные решения (ADR)

| ID | Решение | Обоснование |
|----|---------|-------------|
| ADR-1 | **Hybrid API + Web (Playwright)** | HH API не покрывает все действия (тесты, капчи), веб-версия — fallback |
| ADR-2 | **Android App Client ID/Secret** | Официальное приложение не проходит модерацию, декомпиляция даёт легальный доступ |
| ADR-3 | **SQLite для локального хранилища** | Zero-config, переносимость, достаточно для одного пользователя |
| ADR-4 | **Plugin-based CLI (operations/)** | Open/Closed Principle, легко добавлять команды без изменения ядра |
| ADR-5 | **Docker Compose с 3 сервисами** | Разделение ответственности: collector (сбор), bot (уведомления), worker (отклики) |
| ADR-6 | **Multi-profile через подкаталоги** | Изоляция данных, простое масштабирование, нет конфликтов токенов |
| ADR-7 | **Pydantic Settings + JSON конфиг** | Валидация, типизация, поддержка env vars, вложенные структуры |
| ADR-8 | **Отдельные AI-конфиги для разных задач** | Разные модели/промпты/лимиты для фильтрации, писем, капчи |

## 9. Риски и митигации

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| HH изменит API / заблокирует client_id | Средняя | Критическое | Мониторинг чейнджлога, быстрая смена client_id, fallback на web |
| Теневой бан за спам | Высокая | Высокое | Лимиты, рандомизация задержек, прокси, уникальные cover letter |
| Капча не решается AI | Низкая | Среднее | Fallback на ручной ввод, поддержку kitty/sixel |
| SQLite corruption при крэше | Низкая | Среднее | WAL mode, регулярные бэкапы, команда `migrate` |
| Утечка токенов в логах | Низкая | Критическое | Маскирование в логгере, `--profile` изоляция |
